# Web 管理后台页面扩展建议

## 已落地范围

本轮已经落地以下页面与接口：

- 页面：`/admin/overview`、`/admin/users`、`/admin/groups`、`/admin/conversations`、`/admin/sessions`、`/admin/files`
- 管理 API：`/admin/api/session/*`、`/admin/api/overview`、`/admin/api/users/*`
- 新增只读 API：`/admin/api/groups*`、`/admin/api/conversations*`、`/admin/api/sessions`、`/admin/api/files/*`
- 后台专用聊天文件下载：`/admin/api/files/chat/{file_id}/download`

边界保持不变：

- 不改 `POST /upload/avatar`
- 不改 `POST /upload/chat-file`
- 不改 `GET|HEAD /download/chat-file/*`

下面内容保留为这轮扩展时的设计背景和后续演进建议。

## 1. 现状判断

在扩展之前，后台只提供了一个开发者用户管理页：

- 登录页：`/admin/login`
- 管理页：`/admin/users`
- 后台页面由 `src/server/dev_user_admin_page.cpp` 直接拼接 HTML 返回
- 后台 API 当时只覆盖管理员会话和普通用户管理：`/admin/api/session/*`、`/admin/api/users/*`

但从现有服务端能力看，后台可管理的业务对象已经不止“用户”：

- 用户基础资料与 IM 资料：`user_data`、`user_im_profile`
- 好友关系：`friendships`
- 单聊 / 群聊会话：`conversations`、`conversation_members`
- 消息与送达回执：`messages`、`message_receipts`
- 设备 / 登录会话：`user_sessions`
- 头像与聊天文件上传：`/upload/avatar`、`/upload/chat-file`、`/download/chat-file/*`

所以问题不是“页面不够丰富”，而是“后台信息架构还停留在用户表管理，没有覆盖服务端真实业务面”。

## 2. 设计目标

建议把后台从“单功能页”升级成“小型运维控制台”，目标分成三层：

1. 看全局
   能快速知道当前服务健康度、用户规模、在线情况、群聊规模、消息规模。
2. 查问题
   能按用户、群、会话、消息、文件快速检索，定位投诉或异常。
3. 做操作
   能在明确边界内执行安全操作，例如禁用账号、重置密码、查看群成员、追踪消息、核对文件。

## 3. 后台信息架构

建议把后台路由扩展成下面这组：

- `/admin/overview`：总览仪表盘
- `/admin/users`：用户管理
- `/admin/groups`：群组管理
- `/admin/conversations`：会话与消息
- `/admin/sessions`：在线状态与连接
- `/admin/files`：文件与头像中心
- `/admin/admin-users`：管理员账号
- `/admin/audit`：操作审计日志

如果希望控制初期工作量，可以先做前五个，把 `admin-users` 和 `audit` 作为第二阶段。

## 4. 页面建议

### 4.1 总览仪表盘 `/admin/overview`

这是最应该补的页面。它能把现在“只能进用户表”的后台，升级成真正的入口页。

建议布局：

- 第一行：6 个指标卡
  - 总用户数
  - 启用用户数
  - 在线用户数
  - 好友关系数
  - 群聊数
  - 今日消息数
- 第二行左侧：最近活跃用户 / 最近登录用户
- 第二行右侧：最近创建群聊 / 最近文件上传 / 最近异常提示
- 第三行：快捷入口
  - 新建用户
  - 查群
  - 查会话
  - 查文件

建议强调的数据：

- `user_data.status`
- `user_im_profile.is_online`
- `conversations.type`
- `messages.created_at`
- `conversation_members` 聚合成员数

交互设计：

- 指标卡可点击跳转到对应筛选结果页
- 在线用户数点击进入 `/admin/sessions?online=1`
- 群聊数点击进入 `/admin/groups`

实现难度：

- 中等
- 主要是补充聚合查询 API，不涉及复杂写操作

### 4.2 用户管理 `/admin/users`

现有页面可以保留，但建议从“用户列表 + 创建 + 编辑”升级为“用户工作台”。

建议新增模块：

- 用户详情抽屉 / 右侧详情卡
  - 基础信息
  - IM 资料
  - 最后登录时间
  - 最后在线时间
  - 好友数
  - 所在群数量
- 用户关联数据区
  - 最近会话
  - 最近发送消息
  - 最近上传文件
- 风险操作区
  - 禁用账号
  - 重置密码
  - 清理头像

建议补充筛选项：

- 按状态筛选
- 按在线状态筛选
- 按最近登录时间排序
- 按注册时间排序

当前页面已经有不错的视觉基底，建议保留现在的暖色玻璃质感风格，不必换一套设计语言。

### 4.3 群组管理 `/admin/groups`

这是最自然的第二个核心页面，因为群组已经有完整的数据模型和现成查询逻辑。

建议布局：

- 顶部搜索栏
  - 按 `group_numeric_id`
  - 按群名
  - 按群主 user_id
- 中间主表格
  - 群号
  - 群名
  - 群主
  - 成员数
  - 是否包含当前检索用户
  - 最近更新时间
- 右侧详情面板
  - 群公告
  - 群头像
  - 成员列表
  - 管理员 / 群主角色
  - 最近消息预览

建议操作：

- 查看成员列表
- 复制群 UUID / 群号
- 查看该群最近 20 条消息
- 标记“重点关注群”

暂时不建议一开始就开放高风险写操作，例如“解散群”“踢成员”。当前后台还没有审计体系，先以读和诊断为主更稳。

### 4.4 会话与消息 `/admin/conversations`

这个页面的价值非常高，因为一旦客户端反馈“收不到消息”“消息丢了”“群里发了但没送达”，排查入口都在这里。

建议分成左右两栏：

- 左栏：会话检索列表
  - 按用户 ID / numeric_id 查关联会话
  - 按会话 UUID 查精确会话
  - 按群号 / 群名查群会话
- 右栏：消息时间线
  - 消息序号 `seq`
  - 发送者
  - 消息类型
  - 内容 JSON 摘要
  - 创建时间
  - 送达人数 / 未送达人数

建议关键能力：

- 查看会话基础信息
- 分页拉取最近消息
- 查看某条消息的送达回执
- 过滤系统消息 / 文本消息 / 文件消息
- 通过 `client_msg_id` 或 `message_uuid` 定位单条消息

这个页面本质上是“后台版聊天诊断器”，对排查协议、数据库、在线状态问题都很关键。

### 4.5 在线状态与连接 `/admin/sessions`

现有用户管理页里已经展示了 `is_online`，但这还不够。运维真正关心的是“谁在线、在线多久、从哪里连、是否异常频繁上下线”。

建议页面分成两个 Tab：

- `在线用户`
  - 用户 ID / numeric_id / 用户名
  - 当前是否在线
  - 最后在线时间
  - 活跃会话数
- `连接明细`
  - WebSocket 连接建立时间
  - 远端地址
  - 认证用户
  - 最近收包 / 发包时间
  - 当前连接状态

建议视觉上使用状态色：

- 在线：绿色
- 空闲：灰色
- 异常抖动：红色或橙色

说明：

- 第一层“在线用户列表”可以直接基于数据库字段和内存在线状态实现
- 第二层“连接明细”需要把 `websocket_session` 的内存态做成只读 API

### 4.6 文件与头像中心 `/admin/files`

项目已有头像上传和聊天文件上传下载链路，但后台还没有对应的可视化入口。

建议拆成两个区域：

- `头像文件`
  - 用户 ID
  - 头像 URL
  - 上传时间
  - 文件大小
  - 预览
- `聊天文件`
  - 文件名
  - 发送者
  - 所属会话
  - 上传时间
  - 下载次数或最近访问时间
  - 本地路径 / 下载地址

建议操作：

- 预览头像
- 下载聊天文件
- 根据文件反查消息与发送者
- 查找孤儿文件
  - 磁盘存在但数据库或消息引用不存在

这个页面很适合后期做“磁盘占用分析”和“垃圾文件排查”。

### 4.7 管理员账号 `/admin/admin-users`

当前后台管理员体系已经独立存在，但页面还没有体现。

建议展示：

- 管理员用户名
- 显示名
- 状态
- 最后登录时间
- 创建时间

建议操作：

- 新建管理员
- 禁用管理员
- 重置管理员密码

但如果暂时不准备做角色权限模型，这个页面可以先只读。

### 4.8 操作审计 `/admin/audit`

只要后台开始有“改数据”的能力，这个页面就会变得必要。

建议记录：

- 谁操作的
- 操作时间
- 操作对象
  - 用户
  - 群
  - 管理员
  - 文件
- 操作类型
  - 创建
  - 更新
  - 重置密码
  - 禁用
- 操作前后摘要

如果不做这页，后台功能越多，后续越难追踪误操作。

## 5. 页面框架设计

建议不要继续为每个页面都单独堆一个“完整大卡片”。更适合的是做一个统一后台壳：

- 左侧固定导航
  - 总览
  - 用户
  - 群组
  - 会话
  - 在线
  - 文件
  - 管理员
  - 审计
- 顶部状态栏
  - 当前管理员
  - 当前环境
  - 刷新按钮
  - 退出登录
- 主内容区
  - 页面标题
  - 筛选条
  - 表格 / 详情区 / 时间线区

建议保留现有视觉特征：

- 米白色背景
- 橙色主强调
- 青绿色辅助强调
- 大圆角、半透明面板、轻阴影

这套风格已经比常见“朴素后台”更有识别度，继续沿用可以让新页面自然连贯。

## 6. 实现优先级

### 第一阶段：投入产出比最高

1. `/admin/overview`
2. `/admin/groups`
3. `/admin/conversations`

原因：

- 这三页能最快把后台从“用户表工具”升级成“服务端控制台”
- 都以查询和诊断为主，风险低
- 与现有 IM 核心能力直接对齐

### 第二阶段：补齐运维链路

1. `/admin/sessions`
2. `/admin/files`

原因：

- 这两页对排查在线问题、文件问题很关键
- 但需要补更多运行时状态或本地文件元数据 API

### 第三阶段：进入规范化后台

1. `/admin/admin-users`
2. `/admin/audit`

原因：

- 这部分更偏“后台自身治理”
- 最好放在有更多管理动作之后建设

## 7. 针对当前代码结构的落地建议

由于当前后台页面是直接写在 `src/server/dev_user_admin_page.cpp` 里的原始 HTML 字符串中，建议不要一次性推倒重来，优先做这三件事：

1. 先把“后台壳”和“具体页面内容”拆开
   - 例如拆成 `build_admin_layout(...)`
   - 再分别生成 `overview/users/groups/conversations` 页面主体
2. 先补 GET 型只读 API
   - 先把查询页做起来，再决定哪些写操作真的需要开放
3. 后续再考虑引入更完整的静态资源组织方式
   - 例如把 CSS / JS 从 C++ 字符串中拆到独立静态文件

这样改动成本最低，也最符合现在这个项目的阶段。

## 8. 最推荐的一版最小后台

如果只做一轮增强，我最推荐的版本是：

- 左侧导航 + 顶部状态栏
- `总览`
- `用户管理`
- `群组管理`
- `会话与消息`

这 4 个页面已经足够支撑：

- 日常用户维护
- 群聊问题排查
- 消息链路排查
- 基本服务状态观察

相比继续往当前 `/admin/users` 里硬塞更多卡片，这种结构会更清晰，也更容易继续扩展。
